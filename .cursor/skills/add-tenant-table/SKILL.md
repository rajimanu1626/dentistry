# Add a tenant-scoped table

Multi-tenancy in clinic-crm is enforced by Postgres RLS, not by application
code filters. Every new domain table that holds per-clinic data MUST go through
this checklist to keep the safety guarantee.

## Inputs to gather

1. **Table name** (plural, snake_case: `treatments`, `lab_orders`).
2. **Columns** with types + nullability.
3. **Foreign keys**: at minimum `clinic_id → clinics(id)`, usually also
   `patient_id → patients(id)` or `visit_id → visits(id)`.
4. **Should it be audited?** Most clinical tables should be.
5. **Special indexes** (per-clinic search, time-ordered, etc.).

## Checklist

1. **Add the SQLAlchemy model** in `apps/api/app/models/tables.py`:
   - `id: Mapped[UUIDPk]`
   - `clinic_id: Mapped[UUID]` with `ForeignKey("clinics.id", ondelete="CASCADE")`
   - `created_at: Mapped[CreatedAt]`, optional `updated_at: Mapped[UpdatedAt]`
   - Indexes via `__table_args__`.
   - Re-export from `apps/api/app/models/__init__.py`.
2. **Generate the migration** with `make db-revision m="add <table>"` and
   then **edit it** to include the security boilerplate (the autogenerated
   migration won't include any of this):

   ```python
   def upgrade():
       op.create_table(...)
       op.execute("ALTER TABLE <table> ENABLE ROW LEVEL SECURITY;")
       op.execute("ALTER TABLE <table> FORCE ROW LEVEL SECURITY;")
       op.execute('''
           CREATE POLICY p_<table>_all ON <table> FOR ALL
           USING (clinic_id IN (
             SELECT clinic_id FROM clinic_members
             WHERE user_id = nullif(current_setting('app.current_user_id', true), '')::uuid
           ))
           WITH CHECK (clinic_id IN (
             SELECT clinic_id FROM clinic_members
             WHERE user_id = nullif(current_setting('app.current_user_id', true), '')::uuid
           ));
       ''')
       # If the table's rows belong to a patient and should follow patient_shares:
       # add an additional SELECT policy with the share clause (see patients/visits in 0001_initial).
       op.execute('''
           CREATE TRIGGER audit_<table>
           AFTER INSERT OR UPDATE OR DELETE ON <table>
           FOR EACH ROW EXECUTE FUNCTION audit_row_change();
       ''')
   ```

3. **Add a factory** in `apps/api/tests/factories.py` (create the file the
   first time) so other tests can build rows without rebuilding the FK chain.
4. **Add a focused RLS isolation test** in `apps/api/tests/rls/test_rls_<table>.py`
   modelled on `test_rls_patient_isolation.py`:
   - Two clinics, two users, two rows.
   - User A's session can see only clinic A's row, user B only B's.
   - If applicable, an active `patient_share` reveals the row across clinics.
5. **Update CLAUDE.md** if the table introduces a new domain concept.
6. **Run** `make portability` and `make test-api`.

## Anti-patterns to call out

- Filtering by `clinic_id` in service code instead of relying on RLS.
- Forgetting `FORCE ROW LEVEL SECURITY` — without it, the table owner bypasses
  RLS.
- Using `auth.uid()` in a policy — banned, see portability-invariants.
- Putting PHI in plaintext columns — encrypt via `pgp_sym_encrypt` like
  `patients.phone_enc`.
